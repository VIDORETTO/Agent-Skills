# Onboarding User Activation Optimizer

## Overview

Review onboarding as the path from signup to first meaningful outcome, not as a collection of welcome screens.
Prioritize speed to first value, clarity of next steps, and removal of early confusion or setup drag.

## Operating Mode

Treat onboarding, initial setup, and activation as one continuous experience.
Inspect what the user must understand, configure, import, and accomplish before the product starts feeling useful.

Focus on whether a new user can answer these questions quickly:

- what this product will do for me
- what I need to do first
- how far along I am
- what action will unlock the first useful result
- what happens if I do not yet have data, integrations, or content

Do not stop at walkthrough overlays. Check whether the product genuinely reduces the distance between entry and first value.

## Audit First

Before proposing changes, identify:

1. The product's first meaningful value moment.
2. The minimum actions required to reach it.
3. The setup steps that are truly necessary versus nice-to-have.
4. The highest-friction points where users are likely to hesitate, stall, or abandon.

Use real flows and UI to answer:

- What does the user see immediately after signup or first login?
- Is the next action obvious?
- Are setup steps ordered by necessity or by internal implementation convenience?
- Can the user progress without perfect data or full configuration?
- Do empty states guide action or just reflect absence?
- Is there a visible sense of progress toward activation?

## Review Axes

### 1. Time to first value

Check how quickly the user can experience something useful.

Look for:

- too many prerequisite steps before any payoff
- setup flows that front-load complexity
- early screens that explain too much before the user acts
- required fields or tasks that are not essential for initial value
- first-use experiences that feel administrative rather than outcome-oriented

Prefer flows that produce useful output or insight as early as possible, even if setup continues later.

### 2. Setup sequencing

Review the order of onboarding steps critically.

Check:

- whether prerequisites are truly prerequisites
- whether complex setup is postponed until after initial motivation exists
- whether integrations, imports, and profile completion happen in the right order
- whether users can skip, defer, or partially complete optional steps safely

Flag onboarding that is ordered around internal data dependencies instead of user momentum.

### 3. Guidance and clarity

Inspect how the product teaches the user what to do.

Look for:

- vague onboarding copy
- generic CTAs like "Continue" with little context
- missing rationale for requested setup
- long explanations where one precise sentence would work better
- guidance that tells rather than shows

Prefer guidance that answers:

- why this step matters
- what the user gets from it
- what happens next

### 4. Checklists and progress systems

Treat activation checklists as behavioral tools, not decoration.

Check:

- whether checklist items map to real activation milestones
- whether progress is visible and motivating
- whether items are atomic enough to complete confidently
- whether completed steps unlock a sense of advancement
- whether the checklist disappears, adapts, or evolves once activation is reached

Flag checklists that are busywork, disconnected from value, or overly long.

### 5. Embedded tutorials and product education

Review guided education as support for action, not content for its own sake.

Inspect:

- inline guidance
- tours
- tooltips
- contextual help
- examples shown at the moment of need

Prefer education that appears where friction happens and directly helps the next action.
Flag tutorial systems that interrupt flow or overwhelm users before they have context.

### 6. Initial import and data bootstrap

Check how the product handles the "empty account" problem.

Look for:

- import flows that are too technical for first-time users
- no sample data or no guided alternative when real data is unavailable
- imports that feel risky or irreversible
- long waits without feedback
- users landing in empty dashboards without a recovery path

Prefer bootstrap strategies that reduce fear and help users see value before everything is fully connected.

### 7. Empty states and first actions

Review empty states as activation surfaces.

Check:

- whether empty screens explain the benefit of filling them
- whether the primary action is obvious
- whether the user knows what good looks like
- whether first actions are the highest-value actions or just the easiest to implement

Treat empty states as one of the most important onboarding surfaces in the product.

## Prioritization Rules

Rank issues in this order:

1. Friction that delays or prevents first value.
2. Missing guidance on the next critical action.
3. Setup sequencing that slows momentum unnecessarily.
4. Weak empty states, imports, or progress signals that cause abandonment.
5. Secondary polish around tutorials, wording, or presentation.

Avoid adding more onboarding layers if the real problem is that the underlying path to value is too long or too unclear.

## Output Expectations

When delivering a review, provide:

1. the main activation blockers
2. the root cause behind each blocker
3. the impact on time-to-value or abandonment risk
4. the smallest high-leverage correction
5. any later-stage onboarding refinement worth sequencing after the core fix

When delivering implementation, prefer changes that reduce friction, clarify the next step, and create visible progress toward first value.

## Review Standard

Hold the onboarding experience to this bar:

- the first meaningful outcome is clear
- the next action is obvious
- setup asks are proportionate to value
- progress toward activation is visible
- empty states teach and direct
- users can keep moving even with incomplete setup

If a motivated new user can still stall because the product does not show what to do next or asks too much too early, treat that as a real product failure.
